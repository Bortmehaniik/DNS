# DNS

Здесь показана поэтапная настройка и тестирование DNS сервера в архитектуре master/slave на Debian. Цель преследуемая в этой лабораторной
обеспечить надежное и воспроизводимое развертывание авторитетных DNS-зон с репликацией между *primary (master)* и  *secondary (slave)* серверами, а так же продемонстрировать
практические приемы управления, отладки и обеспечение безопасности.

DNS-зона - это часть пространства имен DNS, за которую отвечает конктретный DNS-сервер и в которой хранится записи (records), описывающие соответсвия имен доменов и различных ресурсов
(ip-адресов, почтовых серверов и т.д.). По сути, зона - это логическая "карта" домена и его поддоменов, оформленная в виде набора DNS-записей.

Есть всего несколько типов DNS-зон:

+ прямая (*forward*) зона - связывает доменные имена с IP-адресами (A/AAAA)
+ обратная (*reverse*) зона - связывает IP-адреса с доменными именами (PTR), используется для обратного разрешения
+ статическая и динамическая - статические файлы зон или динамически изменяемые (DDNS)

Структура файла зоны
Зона хранится в текстовом файле (или в базе конфигурации сервера)

+ SOA (Start of Authority) - основная запись зоны с информацией о владельце зоны, email администратора, серийном номере зоны, интервалы обновление/ретариев и т.д.
+ NS - указывает авторитетные имена серверов для зоны
+ A/AAAA - адресные записи (IPV4/IPV6)
+ CNAME - псевдонимы (canonical name)
+ MX - почтовые сервисы
+ PTR - обратное соответсвие (в обратных зонах)
+ TXT - текстовые записи
+ SRV - сервисные записи
+ TTL - время жизни записей (как долго кешировать ответ)

Есть всего 2 типа сервера, это *Master (primary)* и *Slave (secondary)*, зачастую мастре обладает новыми файлами, а слейв запрашивает новые файлы у мастера.

Что касается безопасности, то есть технология DNSSEC, эта технология позволяет подписывать зоны *ЭЦП*, требует генерации ключей, подписания зоны и публикации DS в родительской зоне.

Перейдем к настройки самого DNS на виртуальных машинах.
Для начала необходимо установить все пакеты связанные с доменными именами, это может быть как просто "bind" так и "bind9utils" и т.д. устанавливаем определенные пакеты в зависимости от выбранного вами дистрибутива Linux.

![image](https://github.com/Bortmehaniik/DNS/blob/image/apt-get%20install%20DNS.png)

Далее необходимо зайти в файл "/etc/bind/named.conf", и добавить туда собственные записи в соответсвии с нашими требованиями, а именно нужно дополнительно указать параметр *allowed-transfer*, данный параметр позволяет обращаться к другому серверу. См. рисунок ниже

![iamge](https://github.com/Bortmehaniik/DNS/blob/image/zone%20from%20default-zone.png)

Далее чтобы не поганить оригинальные конфигурационные файлы зон, их необходимо откапировать с помощью команды cp. Далее зайдя непосредственно в прямую зону конфигурации, необходимо все записи *localhost* заменить на соответсвующие доменное имя *ibis.net*, и далее в конце конфигурационного файла необходимо добавить записи относящиеся к прямой зоне, продемонстрировано на рисунке ниже

![image](https://github.com/Bortmehaniik/DNS/blob/image/dns1.ibis.net.png)

Далее аналогичным методом меняем обратную зону, понятное дело что зоны разные поэтому и записи в данной зоне будут отличаться, см. рисунок ниже

![image](https://github.com/Bortmehaniik/DNS/blob/image/looback.png)

Так же чтобы настроить *slave server*, нужно сделать практически все тоже самое за исключением файл *named.conf* будет *type slave*, и в allowed-transfer мы указываем ip адрес мастер сервера. А так зоны будут похожими. Для проверки работоспособности, можно воспользоваться командой "nslookup DNS", и вместе dns пишем нужное доменное имя.

![iamge](https://github.com/Bortmehaniik/DNS/blob/image/nslookup%20from%20master%20srv.png)

Далее для окончательной проверки работоспособности необходимо создать виртуальную машину с ОС Windows 10, настроить ipv4, и через командную строку так же проверить работает dns или нет, для чистоты проверки можно в конце команды вписать ip адрес нужного сервера для проверки обоих серверов, см. рисунок ниже

![image](https://github.com/Bortmehaniik/DNS/blob/image/proverka%20na%20win(FINISH).png)

Если столкнулись с проблемой на предпоследнем этапе, то стоит прописать свой собственный сервер в файле *resolv.conf*, по аналогии есть похожий файл на windows
